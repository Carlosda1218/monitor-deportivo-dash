CombatIQ

Technical Overview
Versión 1.0 · Mayo 2026

Plataforma de Monitoreo para Deportes de Combate

1. System Overview

CombatIQ es una plataforma web de monitoreo del rendimiento diseñada para deportistas y coaches de artes marciales de combate (Taekwondo y Boxeo). Integra datos biomédicos de sensores, check-ins de bienestar subjetivo e inteligencia artificial en un único sistema multi-rol accesible desde cualquier navegador.

Funcionalidades Principales

  • Monitoreo de señales biomédicas — análisis de ECG/HRV (frecuencia cardíaca y variabilidad) e IMU (detección de golpes, patadas e impactos recibidos).
  • Check-in de bienestar inteligente — cuestionarios ponderados por deporte que calculan un Wellness Score (0–100) pre-entrenamiento.
  • Dashboard del coach — vista de equipo en tiempo real con alertas automáticas, historial de sesiones y métricas de carga.
  • Análisis de carga interna y externa — ACWR (ratio carga aguda:crónica), readiness HRV, tendencias de volumen IMU y alertas por niveles (ok / warning / danger).
  • Insights narrativos con IA — integración con la API de Claude (Anthropic) que genera análisis de coaching contextualizados al deporte y a los datos del atleta.

3. Componentes del Sistema

app.py

Punto de entrada principal. Inicializa el servidor Flask y la instancia Dash, registra el layout base multi-rol (deportista / coach) y define todos los callbacks reactivos. Gestiona la autenticación, la navegación entre páginas y orquesta las cuatro vistas modulares: SignalsView, SensorsView, CompareView y AnalysisView. También expone los endpoints REST internos (/api/sensor-data, /api/sensor-ping) para ingestión de datos desde dispositivos externos.

db.py

Capa completa de acceso a datos. Define todas las tablas SQLite y ejecuta migraciones versionadas de la v10 a la v210 de forma automática e idempotente al arrancar la app. Expone funciones CRUD para: usuarios, sesiones de entrenamiento, cuestionarios, métricas de sensores (ECG, IMU, EMG, RESP), equipos, mensajes internos, peso, nutrición y recuperación de contraseña mediante tokens temporales.

sensors.py

Catálogo declarativo de los sensores del sistema. Cada entrada define nombre, señales que produce, métricas calculadas, compatibilidad por deporte y ejemplos de hardware. Los sensores activos son:

CódigoSensorDeporteMétricas clave

IMU_WRISTIMU muñeca / guanteTKD · Boxeogolpes totales, golpes/min, intensidad media (g), intensidad máxima (g)
IMU_FOOTIMU tobillo / pieTKDpatadas totales, patadas/min, potencia (g), ratio dom./no-dom.

questionnaires.py

Define 22 preguntas organizadas en 7 grupos temáticos: base, taekwondo, boxeo, competencia, peso y molestia. Cada pregunta tiene un peso numérico (3–14) y una dimensión (positive o risk). El módulo implementa el algoritmo de cálculo del Wellness Score (0–100): normaliza cada respuesta al rango 0–100, invierte las dimensiones de riesgo y pondera cada ítem según su peso relativo al sport del atleta.

analysis_engine.py

Motor de análisis que agrega datos históricos del atleta y genera un informe estructurado. Calcula: ACWR (ratio carga aguda:crónica de las últimas 7 vs 28 sesiones), readiness HRV comparada contra la baseline rolling de 30 días, tendencias de volumen IMU y el listado de alertas automáticas clasificadas por nivel (ok, warning, danger). Incluye caché en memoria con TTL de 5 minutos para no recalcular en cada renderizado.

ai_insights.py

Integración con la API de Claude de Anthropic. Recibe el informe del analysis_engine y genera un análisis narrativo de coaching adaptado al deporte y al momento de la temporada. Implementa caché en memoria (10 min), timeout configurable (defecto 8 s) y degradación elegante: si no hay API key o falla la llamada, retorna un mensaje de aviso sin romper la UI.

4. Flujo de Datos

Flujo 1: Check-in de bienestar (deportista)

  1. El deportista autenticado navega a la página /wellbeing.
  2. El layout detecta su deporte (users.sport) y muestra solo las preguntas relevantes (preguntas base + grupo específico del deporte).
  3. El deportista responde los sliders de 1–5 (energía, recuperación, sueño, fatiga, etc.) y registra RPE y duración de la sesión anterior.
  4. questionnaires.compute_wellness_score(answers, sport) normaliza cada respuesta a 0–100, invierte las dimensiones de riesgo y aplica los pesos configurados.
  5. El score final (0–100) y el JSON de respuestas se persisten en la tabla questionnaires junto al user_id, timestamp y session_id.
  6. analysis_engine.invalidate_cache(uid) limpia el caché del atleta.
  7. El dashboard del coach actualiza la vista del equipo: el wellness del atleta aparece con su nivel de color (verde / naranja / rojo).
  8. Si ANTHROPIC_API_KEY está configurada, ai_insights.generate_coaching_note(report, athlete_name, sport) devuelve el análisis narrativo visible para el coach.

Flujo 2: Análisis de señal ECG

  1. El coach o deportista sube el CSV de la banda torácica en la vista Signals.
  2. El backend lee el archivo, detecta la frecuencia de muestreo y registra el archivo en ecg_files.
  3. analysis_engine carga las muestras y aplica scipy.signal.find_peaks() para detectar los picos R.
  4. Se calculan los intervalos R–R y a partir de ellos: BPM, SDNN y RMSSD.
  5. Las métricas se guardan en ecg_metrics vinculadas al ecg_file_id y session_id.
  6. La vista Signals renderiza la señal ECG en tiempo real con los picos marcados y las métricas en tarjetas de resumen.

Relación entre tablas principales

Un usuario (role=deportista) pertenece a un coach vía coach_athletes (relación de adopción) o por el campo legacy users.coach_id. Cada session agrupa las métricas de la sesión: enlaza con ecg_files, imu_metrics, questionnaires y sensor_sessions mediante session_id.

6. Procesamiento Biomédico

Señal ECG (electrocardiograma)

ParámetroSignificadoCálculo
BPM Frecuencia cardíaca (latidos por minuto) Se detectan los picos R con scipy.signal.find_peaks() sobre la señal filtrada. BPM = 60 / media(intervalos R–R en segundos).
SDNN Variabilidad de la frecuencia cardíaca (HRV global) Desviación estándar de todos los intervalos R–R de la grabación. Valores bajos indican fatiga acumulada.
RMSSD HRV de corto plazo (actividad parasimpática) Raíz cuadrada del promedio de los cuadrados de las diferencias entre intervalos R–R consecutivos. Indicador de recuperación autonómica.
n_peaks Total de latidos detectados Contador de picos R identificados en el segmento analizado. Sirve para validar la calidad de la señal.

Señal IMU (unidad de medición inercial)

ParámetroSignificadoCálculo
n_hits Número total de golpes / patadas / impactos Se calcula la magnitud vectorial de la aceleración: √(ax² + ay² + az²). Los picos que superan un umbral de detección (≥ 2 g) se cuentan como impactos.
hits_per_min Cadencia (golpes por minuto) n_hits / duración de la grabación en minutos. Indicador de volumen de trabajo específico.
mean_int_g / max_int_g Intensidad media y máxima del impacto Valor de pico de la magnitud vectorial en cada impacto detectado, expresado en g (gravedades). La media mide la intensidad sostenida; el máximo detecta golpes de alta potencia.
mean_ang_vel / max_ang_vel Velocidad angular media y máxima Derivada del giroscopio triaxial. Cuantifica la rapidez de rotación del segmento corporal durante el impacto (relevante para técnicas giratorias en TKD).
Nota: La ingestión de sensores acepta tres modalidades: (1) carga manual de CSV/JSON desde la UI, (2) POST al endpoint /api/sensor-data desde hardware externo, y (3) BLE a través de un hub Nordic UART (en desarrollo activo).

7. Limitaciones

  1. Base de datos SQLite: concurrencia limitada bajo cargas altas simultáneas (múltiples coaches + atletas en la misma instancia). El modo WAL mejora la situación pero no la elimina.
  2. Sin integración BLE en tiempo real: la ingestión de ECG e IMU es mayoritariamente manual (CSV/JSON upload). El hub BLE Nordic está en fase de prototipo.
  3. Sin app móvil nativa: la plataforma es solo web (PWA instalable). No existe versión iOS o Android en tiendas de aplicaciones.
  4. Insights de IA opcionales: requieren ANTHROPIC_API_KEY activa. Sin ella la sección de análisis narrativo no se muestra, pero el resto del sistema funciona sin modificaciones.
  5. Análisis biomecánico por video (YOLO / MediaPipe) es computacionalmente intenso: no es apto para servidores de bajo costo o entornos sin GPU. Las estimaciones de pose se realizan en modo offline sobre archivos de video previamente grabados.

8. Mejoras

Arquitectura

Migrar a una arquitectura API REST + SPA (FastAPI en backend, React o Vue en frontend) para desacoplar las capas y escalar cada una de forma independiente. Introducir Redis como caché compartida para los informes del analysis_engine, eliminando el caché en memoria no compartido entre procesos.

Base de datos

Migrar de SQLite a PostgreSQL cuando el número de atletas activos supere los 300 o cuando se despliegue en producción con múltiples coaches concurrentes. Adoptar SQLAlchemy como ORM para simplificar las migraciones (Alembic) y mejorar la cobertura de tests. Mantener la capa de migraciones versionadas que ya existe en db.py para una transición sin pérdida de datos.

Funcionalidad

Integración BLE en tiempo real con transmisión de datos desde sensores wearables directamente a la plataforma sin pasos manuales de carga. Notificaciones push al móvil del coach cuando un atleta registra un wellness score bajo o un impacto de alto riesgo. App móvil nativa (React Native) para el check-in diario del deportista sin necesidad de abrir el navegador.

9. Conclusiones

Qué has aprendido

  • Cómo construir un sistema web multi-rol real (coach / deportista) con autenticación segura (bcrypt + tokens de sesión), navegación protegida y separación de permisos por rol.
  • Procesamiento de señales biomédicas en Python: detección de picos R en ECG con scipy, cálculo de parámetros HRV (SDNN, RMSSD) y detección de impactos en señales IMU triaxiales.
  • Diseño de un esquema de base de datos evolutivo con migraciones versionadas que permiten añadir funcionalidades en producción sin perder datos existentes.
  • Integración de modelos de IA generativa (Claude API) en el flujo de datos de una aplicación real, con manejo de errores, caché y degradación elegante cuando el servicio no está disponible.
  • Patrones de arquitectura reactiva con Plotly Dash: cómo estructurar callbacks complejos con múltiples inputs/outputs, estados compartidos y componentes dinámicos por deporte.

Qué mejorarías

  • Separar el frontend del backend desde el inicio (API REST + cliente independiente) para facilitar el testing y la escala horizontal.
  • Implementar WebSockets (o Server-Sent Events) para que el dashboard del coach se actualice en tiempo real sin necesidad de recargar la página.
  • Añadir una suite de tests automatizados (pytest + coverage) desde el primer día, especialmente para el algoritmo del Wellness Score y el pipeline de análisis ECG/IMU.
  • Usar PostgreSQL desde el despliegue inicial en lugar de migrar después, para evitar el trabajo de conversión de datos en producción.
